# Оптимизация SQL-запросов

## Общая информация

В рамках тестового задания были разработаны аналитические SQL-запросы для PostgreSQL. Основное внимание уделено повышению производительности запросов за счет использования оконных функций, Common Table Expressions (CTE), индексов, материализованных представлений и рекомендаций по секционированию таблиц.

---

# 1. Оптимизация запроса «Анализ выручки филиалов»

## Используемые технологии

- Common Table Expressions (CTE)
- Window Functions
- `LAG()`
- `DENSE_RANK()`
- Скользящее окно (Rolling Window)

## Выполненная оптимизация

Запрос разбит на несколько CTE:

- `monthly_revenue` — предварительная агрегация данных по филиалам и месяцам;
- `rolling_revenue` — расчет скользящей выручки и предыдущего периода;
- `ranked_filials` — ранжирование филиалов по выручке.

Такой подход уменьшает повторные вычисления и делает запрос более читаемым.

### Рекомендуемый индекс

```sql
CREATE INDEX CONCURRENTLY idx_fact_ops_filial_period
ON fact_operations (FILIAL, PERIOD)
INCLUDE (SUMMA, CNT);
```

### EXPLAIN ANALYZE

При выполнении запроса рекомендуется проверить:

- Planning Time;
- Execution Time;
- использование Index Scan;
- отсутствие полного Sequential Scan.

# 2. Оптимизация RFM-анализа

## Используемые технологии

- `NTILE()`
- CTE
- `CASE`
- `STRING_AGG()`

## Выполненная оптимизация

RFM-анализ разделен на три этапа:

1. Расчет исходных показателей (Recency, Frequency, Monetary);
2. Присвоение рейтингов;
3. Формирование сегментов.

Такое разделение упрощает сопровождение кода и позволяет PostgreSQL эффективнее строить план выполнения.

### Рекомендуемые индексы

```sql
CREATE INDEX idx_fact_operations_date
ON fact_operations (DATA);

CREATE INDEX idx_fact_operations_filial
ON fact_operations (FILIAL);
```

Индекс по дате значительно ускоряет выборку операций за последние 90 дней.

# 3. Оптимизация анализа услуг

## Используемые технологии

- `LAG()`
- Window Functions
- Pareto-анализ
- Накопительная сумма (`SUM() OVER`)

## Выполненная оптимизация

Сначала данные агрегируются по услугам, затем рассчитываются:

- темп роста;
- накопленная доля;
- правило Парето.

Такой подход позволяет избежать повторных вычислений.

### Рекомендуемый индекс

```sql
CREATE INDEX CONCURRENTLY idx_fact_ops_incgk
ON fact_operations (INC_GK)
INCLUDE (SUMMA, PERIOD, FILIAL);
```

Индекс ускоряет:

- анализ услуг;
- соединение со справочником продуктов;
- Pareto-анализ.

## Используемые технологии

## Выполненная оптимизация

Все показатели рассчитываются при помощи оконных функций:

- предыдущая выручка;
- средняя выручка;

# 5. Materialized View

Для ускорения загрузки отчетов рекомендуется использовать материализованное представление.

```sql
CREATE MATERIALIZED VIEW mv_filial_monthly_kpi AS
SELECT
    filial,
    period,
    inc_gk,
    SUM(summa) AS revenue,
    SUM(cnt) AS operations,
    SUM(weight) AS total_weight
FROM fact_operations
GROUP BY filial, period, inc_gk;
```

После выполнения ETL необходимо обновлять представление:

```sql
REFRESH MATERIALIZED VIEW mv_filial_monthly_kpi;
```

## Преимущества

- уменьшение времени выполнения аналитических запросов;
- ускорение загрузки Power BI;
- снижение нагрузки на основную таблицу.

# 7. Стратегия секционирования

При увеличении объема данных рекомендуется использовать секционирование таблицы `fact_operations`.

Тип секционирования:

- **Range Partitioning**

Ключ секционирования:

- `DATA`

Пример структуры:

```text
fact_operations
├── 2024
├── 2025
├── 2026
└── ...
```

### Преимущества

- ускорение выборок по датам;
- уменьшение размера индексов;
- упрощение архивирования данных;
- повышение производительности запросов.
